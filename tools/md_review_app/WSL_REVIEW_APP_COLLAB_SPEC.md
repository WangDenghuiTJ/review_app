# WSL + Review App 协作规范

## 1. 适用场景

这份规范适用于下面这类工作流：

- 智能体在 `WSL` 内生成较长的计划书、方案说明、执行规划或分析文档
- 文档写入 `MT Review App` 实际加载的 Markdown 文件
- 用户在浏览器里审阅、批注、追问、要求修改
- 智能体读取评论线程和正文，继续补充、修订、回复
- 用户确认后，再让智能体按最终方案推进执行

这套流程的核心是“文件驱动协作”，不是“浏览器驱动协作”。

结论：

- 智能体主要读写 Markdown 和评论 JSON
- 用户主要在浏览器里做审阅
- 浏览器不需要成为主数据源

## 2. 推荐环境结构

推荐的工作结构如下：

- `WSL`：运行智能体、脚本、服务端、正文生成与后续执行逻辑
- `Windows 浏览器`：打开 Review App 页面，负责人工审阅和批注
- `Review App`：作为“审阅界面 + 线程化批注中间层”

推荐方式：

1. 在 `WSL` 内运行 `server.py`
2. 在 `Windows` 上用 `Chrome` 或 `Edge` 打开 `http://127.0.0.1:8765`
3. 智能体在 `WSL` 内直接读写正文与评论文件

不建议把“让智能体依赖浏览器点击页面”作为主链路。

## 3. 当前系统中的关键文件

按当前项目实现，默认工作区根是：

- `E:\code\PyCharm`

默认正文文件是：

- `E:\code\PyCharm\workflow_review.md`

默认评论文件是：

- `E:\code\PyCharm\workflow_review.md.comments.json`

项目代码位于：

- `E:\code\PyCharm\tools\md_review_app`

重要约束：

- `tools\md_review_app\workflow_review.md` 不是默认正文
- 真正要给用户审阅的内容，应写到工作区根目录下的 `workflow_review.md`

## 4. 角色分工

### 用户负责

- 提出目标、范围和约束
- 在 Review App 中阅读正文
- 对正文进行批注、追问、要求修改
- 对最终方案做确认或否决

### 智能体负责

- 在 `WSL` 内生成初稿
- 把初稿写入目标 Markdown 文件
- 读取评论线程并回复
- 根据批注更新正文
- 在方案确认后推进执行

### Review App 负责

- 展示正文
- 承载批注和线程回复
- 将评论独立保存到 `.comments.json`

## 5. 固定操作流

### 阶段 A：建立任务

目标：明确本轮要产出的文档是什么。

标准动作：

1. 用户向智能体说明任务目标、边界和预期产物
2. 智能体确认目标文档应写入 `workflow_review.md`
3. 智能体判断是“覆盖旧稿”还是“在旧稿基础上修订”

建议要求：

- 明确文档类型，例如“计划书”“执行方案”“调研结论”“路线图”
- 明确是否允许保留旧版痕迹
- 明确本轮是否只定稿，不执行

### 阶段 B：生成初稿

目标：让用户先有一个可审阅的完整版本。

标准动作：

1. 智能体在 `WSL` 内生成初稿
2. 智能体把正文写入 `workflow_review.md`
3. 如有必要，清理或保留旧评论文件
4. 用户在浏览器里加载页面并开始审阅

初稿要求：

- 结构完整
- 标题清晰
- 术语统一
- 不要求第一次就完全正确

### 阶段 C：用户审阅

目标：把“问题”沉淀成线程，而不是散落在聊天里。

标准动作：

1. 用户在正文上添加选区批注或行级批注
2. 用户对关键问题使用“标记待处理”
3. 如果有多个问题要一起处理，可使用“标记全部待处理”
4. 必要时使用“复制待处理”生成待处理线程摘要

审阅规则：

- 一条批注只表达一个问题
- 如果是改写要求，应明确“改什么”和“为什么改”
- 如果是追问，应明确希望补充的数据、口径或论证

### 阶段 D：智能体处理批注

目标：让智能体以线程为单位回复，而不是口头承诺。

标准动作：

1. 智能体读取 `workflow_review.md.comments.json`
2. 优先处理 `assistantRequest.requested = true` 且未响应的线程
3. 对每个线程给出明确回复
4. 如批注要求正文修改，则同步更新 `workflow_review.md`
5. 回复完成后，将线程状态更新为已回复

处理原则：

- 先回复线程，再改正文，或在回复中明确“已按此修改正文”
- 不要只在聊天里说“我改了”，却不写回评论文件
- 不要跳过待处理线程直接改正文，否则用户很难核对

### 阶段 E：定稿确认

目标：把“审阅中”切换为“已确认”。

标准动作：

1. 用户检查正文与线程回复
2. 用户对已解决问题标记“已解决”或明确表示通过
3. 智能体确认当前版本可作为执行依据
4. 必要时归档当前正文与评论快照

定稿标准：

- 正文结构完整
- 关键批注已有明确回复
- 争议点已关闭或单独列为待决事项
- 用户明确允许进入执行阶段

### 阶段 F：按定稿执行

目标：避免“边审边做”导致执行口径漂移。

标准动作：

1. 智能体把当前已确认方案视为唯一执行依据
2. 后续执行如果偏离方案，应先提出变更理由
3. 若执行中需要追加规划，重新进入新的审阅循环

## 6. 文件级约定

### 正文文件约定

- 正文文件是单一事实来源
- 方案正文必须落在 `workflow_review.md`
- 智能体改文时，应优先保持标题层级和章节结构稳定

### 评论文件约定

- 评论线程保存在 `workflow_review.md.comments.json`
- 线程消息应保留完整上下文，不要手工删历史消息
- 用户消息默认应为 `role=user`
- 智能体消息默认应为 `role=assistant`

### 状态约定

线程至少有三种工作状态：

- `待处理`：用户明确要求智能体处理
- `进行中`：线程存在但未挂待处理
- `已解决`：用户确认问题关闭

推荐处理顺序：

1. 待处理
2. 进行中
3. 已解决

## 7. 对话规范

为了让智能体更稳定，建议用户在聊天里使用下面三类指令。

### 7.1 新建初稿

可直接这样说：

```text
基于下面要求，生成一版完整方案，并写入 MT Review App 默认加载的 workflow_review.md，先不要执行，只让我审阅。
```

### 7.2 处理批注

可直接这样说：

```text
读取 Review App 里当前待处理线程，逐条回复；凡是你认同并需要落到正文的，直接同步修改 workflow_review.md。
```

### 7.3 定稿后执行

可直接这样说：

```text
以当前 workflow_review.md 的已确认版本为准开始执行。如果发现执行口径需要偏离，先列出偏离点，再继续。
```

## 8. 推荐的会话策略

### 策略 1：规划与执行分离

建议把“写方案”和“按方案做事”分成两个明确阶段。

原因：

- 减少智能体边写边做造成的漂移
- 用户更容易审阅
- 后续执行的依据更清晰

### 策略 2：评论线程优先，不靠散聊记忆

用户对正文的意见，应尽量留在 Review App 线程里，而不是散落在聊天记录里。

原因：

- 线程可回溯
- 能绑定具体原文位置
- 更适合多轮修改

### 策略 3：重大修改必须回写正文

如果智能体说“已采纳”，那就应该同步把正文改掉。

不推荐的做法：

- 聊天里口头承诺已修改
- 但正文未更新
- 评论线程也没有留处理结果

## 9. 什么时候需要 WSL 内浏览器

大多数情况下，不需要。

只有在下面情况，才建议在 `WSL` 里安装 `Chromium` 或 `Chrome`：

- 需要自动化点页面
- 需要做前端交互测试
- 需要截图、录制或 DevTools 调试
- 需要让智能体直接检查浏览器实际渲染结果

如果只是“智能体写文档 + 用户审阅批注 + 智能体改文”，Windows 浏览器已经足够。

## 10. 常见失败点

### 1. 改错文件

错误表现：

- 智能体改了 `tools\md_review_app\workflow_review.md`
- 用户页面里看不到修改

正确做法：

- 始终确认真正加载的是工作区根目录下的 `workflow_review.md`

### 2. 只改聊天，不改线程

错误表现：

- 智能体在聊天里解释过了
- 但评论线程没有回复

正确做法：

- 以评论线程作为审阅闭环记录

### 3. 只回复线程，不改正文

错误表现：

- 智能体说“已采纳”
- 但正文还停留在旧版本

正确做法：

- 凡是采纳型回复，应同步改正文

### 4. 边审边执行

错误表现：

- 文档还没定稿
- 智能体已经开始做后续实现

正确做法：

- 等用户明确确认后，再进入执行阶段

## 11. 最小执行口径

如果要把这份规范压缩成最短操作口径，只保留下面 5 条：

1. 智能体在 `WSL` 里生成和修改正文
2. 正文统一写入 `workflow_review.md`
3. 用户在浏览器里通过 Review App 审阅和批注
4. 智能体优先处理待处理线程，并把采纳内容同步回写正文
5. 用户确认定稿后，智能体再按最终方案执行
