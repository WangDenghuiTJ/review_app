# 藻华研究后续推进计划

更新时间：2026-04-09

## 一、当前判断

结合 Obsidian 里的项目总控、执行树和模块页，当前最合理的推进顺序不是同时铺开三篇论文，而是先把论文1的核心输入和方法底座做实，再把共享基础补到“够用且可复用”的程度。

- 主线任务：`论文1 / M02 反演模型`
- 并行支撑：`S02A GOCI-II 下载`、`S03B/S03C 气象与风场输入`、`S01 漂移追踪方法 API 化`
- 暂缓事项：`论文2 短期预测系统`、`论文3 全球漂移预测` 的系统级展开

当前已经具备的基础是：

- `M01 现象证据` 已基本成立，已经能支持“高温后表层指标下降”这条现象线。
- `M02` 的方法框架已经明确，但真正卡住的是输入口径、样本构建规则和验收标准还没有完全固化。
- 共享基础里已经有若干脚本和链路，但还没有统一成“稳定输入 + 稳定接口 + 稳定状态登记”。

## 二、总体推进原则

### 1. 先做论文1闭环，不抢跑论文2/论文3

在现有笔记结构下，论文2和论文3都明确依赖 `M02` 和共享基础。现在如果提前去做系统设计，产出大概率会停留在占位层，反而会稀释主线推进。

### 2. 优先把 `M02` 做成可复用产品，而不是只跑出一次结果

后续真正有价值的不是“某次训练出了一个结果”，而是形成一套能稳定复用到 `M04`、`M05`、论文2 的输入产品和方法说明。

### 3. 共享基础只做“当前主线真正需要的最小闭环”

共享基础很重要，但近期不需要把所有数据线都铺满。应优先补齐那些直接影响 `M02`、并且能顺手为论文2/3留下标准接口的部分。

## 三、建议推进节奏

## 阶段 A：先把 `M02` 的输入契约和样本构建规则写死

建议时长：1 到 2 周

目标：让 `M02` 从“方法想法明确”变成“输入、流程、输出都可执行”。

本阶段应完成：

- 汇总 `D01` 全部输入来源，明确太湖、巢湖当前可直接用于 `M02` 的数据范围。
- 汇总 `D02` 全部变量、字段、时间范围、单位和缺失情况，形成统一输入表。
- 固化像素对齐流程，明确使用什么配准方法、哪些情况判为失败、失败样本如何处理。
- 固化 `t_max` 筛选规则，明确风速阈值、时间窗和冲突情形的处理办法。
- 固化伪标签构建规则，明确 `p` 和 `B` 的计算流程，以及假设来源和局限。
- 把训练集、验证集、测试集的时间划分和样本统计单独列出来，避免后面口径漂移。

本阶段建议交付物：

- 一份 `M02` 输入总表
- 一份样本构建流程说明
- 一份训练配置与数据划分说明
- 一个“太湖单湖闭环样例”跑通记录

这里建议优先以太湖作为闭环对象，因为现有笔记里太湖的数据完整性明显高于其他湖泊。

## 阶段 B：跑出 `M02` 第一版稳定结果，并建立验收口径

建议时长：2 到 3 周

目标：让 `M02` 稳定输出 `p` 和 `B`，并且这些输出能直接供后续分析使用。

本阶段应完成：

- 固化预训练配置，避免后续每次换一套参数。
- 输出表层占比 `p` 的初版结果，并检查空间分布、范围约束和异常值。
- 输出水柱总量 `B` 的初版结果，并检查与表层指标是否存在明显不合理偏离。
- 明确评价指标，包括拟合误差、稳定性、失败样本比例和主要局限。
- 形成一个“结果是否可供 `M04` / `M05` 复用”的验收判断。

本阶段建议交付物：

- 初版 `p` 产品
- 初版 `B` 产品
- 验收说明文档
- 错误案例清单

这一阶段不要急着进入 PINN 微调。先确认预训练链路和初版结果能被分析模块稳定消费，再决定 `M03` 的优先级会更稳。

## 阶段 C：用 `M02` 结果快速推进 `M04` 和 `M05`

建议时长：1 到 2 周

目标：尽快把“表层下降不等于总量同步下降”从概念主线推到结果主线。

本阶段应完成：

- 在 `M04` 中定义日尺度 `p` 汇总方式，并先出 `p-T` 关系图。
- 在 `M05` 中定义日尺度 `B` 汇总方式，并先出 `B-T` 关系图。
- 设计一个“表层-总量分离”指标，直接表达二者在高温阶段是否脱钩。
- 先出最小图组，而不是一次性追求完整论文图版。

本阶段建议最先产出的图：

- `p-T` 曲线或箱线图
- `B-T` 曲线或箱线图
- 表层指标与总量指标对比图
- 一个高温阶段的代表性时空案例图

如果这一步做顺，论文1 的主线就会从“现象证据”进入“机制闭环雏形”。

## 阶段 D：并行补齐共享基础，但只做与主线直接相关的部分

建议时长：与阶段 A 到 C 并行推进

### `S02A GOCI-II 下载`

近期只需要把这条线推进到“可批量补数”：

- 填入真实 URL 前缀
- 跑通首轮 URL 探测
- 执行首轮下载
- 下载完成后接入 AFAI 处理流程

### `S03B / S03C 气象与风场输入`

近期重点不是无限扩湖，而是先形成统一口径：

- 启动巢湖 `ERA5-Land` 实际下载
- 复核太湖现有风场和环境数据的字段、时间覆盖和目录结构
- 固化 `HRRR 主驱动 + URMA 验证` 的北美口径
- 统一变量矩阵、时间命名规范和元数据结构

### `S01 漂移追踪方法`

近期只需要推进到“可调用组件”：

- 封装稳定 API 或脚本入口
- 明确输入输出格式
- 固化一个最小可调用示例
- 先设计验证数据集框架，再补精度评估

这部分现在不需要追求单独成稿，先变成稳定轮子更重要。

## 四、接下来最值得立即执行的 6 件事

1. 在 `M02` 下补一份正式的输入总表，把 `D01` 和 `D02` 的路径、字段、时间范围、单位一次写清。
2. 用太湖数据跑一个小范围样例，把像素对齐、`t_max` 选择和伪标签构建完整走一遍。
3. 固化 `M02` 的训练配置和验收标准，避免边训练边改口径。
4. 启动巢湖 `ERA5-Land` 下载，并同步检查太湖现有环境数据是否已经满足 `M02` 输入要求。
5. 把 `S02A` 的真实 URL 前缀补上，开始第一轮 `GOCI-II` 探测和下载。
6. 把 `S01` 封装成最小可调用入口，哪怕先只有一个稳定函数签名和示例脚本。

## 五、明确暂缓的事项

现阶段建议明确不优先做以下事项：

- 不提前细化论文2的三模块耦合设计
- 不提前扩展论文3的全球多湖泊框架
- 不过早把精力放在“是否独立成稿”的归属判断上

原因很简单：这些工作都建立在 `M02` 稳定输出、共享基础可复用之后，现在抢跑，产出不会稳。

## 六、建议的阶段性验收标准

当下面 4 条满足时，可以认为项目进入下一阶段：

- `M02` 已能稳定输出 `p` 和 `B`
- `M04` / `M05` 已能基于 `M02` 结果直接出图
- `S02A` 和 `S03B/S03C` 已形成可重复运行的数据补齐流程
- `S01` 已具备明确接口和最小调用样例

一旦这 4 条成立，就可以正式启动论文2，并开始讨论论文3的标准化扩展路线。
